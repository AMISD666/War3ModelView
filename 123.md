# War3ModelView: 批量模式加载与资源解析技术文档

本文档详细介绍了 War3ModelView 在“批量模式”（Batch Mode）下加载模型、解析结构、解码贴图及其复杂的贴图查找逻辑。

---

## 1. 模型加载与解析流程 (Model Loading & Parsing)

### 1.1 入口与扫描
- **触发**: 用户通过 `BatchManager` 选择文件夹，调用 `scanFolder` 递归扫描目录。
- **匹配**: 识别 `.mdx` (二进制) 和 `.mdl` (文本) 后缀的文件，生成文件列表。
- **异步处理**: 使用 Web Worker 集群并行处理。`ThumbnailService` 负责调度，根据 CPU 核心数启动多个 `thumbnail.worker.ts`。

### 1.2 模型解析
- **库支持**: 使用 `war3-model` 库进行解析。
- **元数据提取**:
  - **动画序列**: 提取序列名称，智能筛选（如优先匹配 "Stand"）用于预览。
  - **贴图列表**: 遍历 `Textures` 区块及粒子发射器中的路径。
  - **边界计算**:  worker 线程会计算模型的 AABB (轴对齐边界框)，用于自动设置相机距离，确保模型在缩略图中完美填充视野。

## 2. 贴图查找逻辑 (Texture Resolution Strategy)

系统采用多层级备选方案搜索贴图，确保护度最高地显示模型。

### 2.1 路径标准化
所有路径均通过 `normalizePath` 处理：
- 转换所有 `/` 为 `\`。
- 移除多余的空字符和双斜杠。
- 移除前导路径。

### 2.2 替换贴图 (Replaceable Textures)
如果贴图具有 `ReplaceableId`，则映射到标准魔兽路径：
- `1` -> `ReplaceableTextures\TeamColor\TeamColor00.blp`
- `2` -> `ReplaceableTextures\TeamGlow\TeamGlow00.blp`
- `11` -> `ReplaceableTextures\Cliff\Cliff0.blp`
- 以及树木、蘑菇等环境贴图。

### 2.3 本地文件系统查找 (FS Search)
按以下优先级在模型所在目录下搜索：
1. **原始相对路径**: `模型所在目录 + 模型内记录的相对路径`。
2. **纯文件名查找**: 忽略内部路径层级，仅尝试 `模型所在目录 + 贴图文件名`。
3. **递归向上查找**: 如果在当前目录未找到，会递归向上搜索父级目录直到驱动器根目录（常用于处理贴图放在模型外层目录的情况）。

### 2.4 游戏数据包查找 (MPQ/CAS)
如果本地未找到，通过 Tauri 命令 `read_mpq_file` 调用 Rust 层从游戏 MPQ/CAS 文件读取：
- 识别标准前缀（如 `Textures\`, `Units\`, `Buildings\`）。
- 即使路径不符合标准，也会进行一次全路径匹配尝试。

## 3. 贴图解码与优化 (Texture Decoding & Optimization)

### 3.1 格式支持
- **BLP**: 魔兽原生格式，支持多级 Mipmap。
- **TGA**: 常用于自定义模型，worker 内置了健壮的 TGA 解码器（支持 RLE 压缩、8/16/24/32位颜色）。
- **PNG**: 本地文件特有的支持。

### 3.2 离线解码优化 (Worker-side Decoding)
- **非预乘 Alpha 保护**: 使用 `createImageBitmap` 时显式设置 `premultiplyAlpha: 'none'`。这是**关键优化**，防止 Canvas 2D 自动混合 Alpha 导致透明区域残留黑色或丢失颜色信息。
- **智能 Mipmap 选择**: 缩略图模式下会根据所需尺寸计算最佳 Mip 等级，节省显存（例如 512x512 贴图在 128 缩略图下可能只解码第 2 级 Mip）。
- **亮度补偿回退 (Luma Fallback)**: 某些自定义模型的低级 Mip 数据损坏（呈现全黑），系统会自动检测解码后的 ImageData 亮度，若低于阈值且几乎不含高亮像素，则强制回退使用顶级 Mip (Mip 0)。

## 4. 批量性能管理

### 4.1 显存管理 (VRAM Control)
- **显存计数器**: 追踪每个贴图路径在 GPU 中的引用计数。
- **按需释放**: 当某个贴图不再被当前缓存的所有模型使用时，立即从 GPU 中卸载该贴图，确保在预览上千个模型时显卡不会崩溃。

### 4.2 缓存亲和性 (Worker Affinity)
- 记录模型路径与 worker 的映射关系。
- 再次渲染同一模型（如切换速度、旋转）时，优先派发给同一个 worker，利用 worker 内部的 WebGL 纹理和模型缓冲区缓存。

### 4.3 渲染隔离
每个 worker 拥有独立的 `OffscreenCanvas` 和 WebGL 上下文，彻底隔离渲染状态，防止复杂的粒子系统或材质 Shader 在主线程发生冲突。
